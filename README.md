# RADICI

RADICI is a cultural heritage exploration platform built around a Flask application, a Redis-backed dataset, and a small supporting data pipeline used to collect, enrich, export, and restore project data.

This repository is organized to support two distinct but connected concerns:

- application runtime
- data preparation and operational maintenance

The runtime serves a map-based and grid-based exploration interface for cultural objects aggregated from multiple archives. The data pipeline provides the scripts needed to ingest source material, enrich records, export frontend-ready datasets, and restore a local Redis instance from a project dump.

## Project Goals

The project aims to:

- aggregate cultural heritage records from multiple archives
- expose those records through a web interface
- support geographic exploration through a map
- support text, keyword, and similarity-based discovery
- support user accounts and personal collections
- keep data processing concerns separate from the web runtime

## System Architecture

### Runtime Architecture

The active runtime is intentionally simple:

1. The Flask application in [backend/app.py](/media/data/giacomo.vallasciani/RADICI/backend/app.py) is the single application entry point.
2. Flask renders HTML templates from [frontend/templates](/media/data/giacomo.vallasciani/RADICI/frontend/templates).
3. Flask serves static assets from [frontend/static](/media/data/giacomo.vallasciani/RADICI/frontend/static).
4. Object records and user/collection data are loaded from Redis.
5. Images are served from [backend/downloaded_images](/media/data/giacomo.vallasciani/RADICI/backend/downloaded_images).
6. The frontend consumes both dynamic API endpoints and a static GeoJSON export.

This means the repository no longer uses separate backend and frontend applications. The frontend is part of the Flask runtime and is delivered directly by the backend.

### Logical Components

The project is best understood as five cooperating layers:

1. Presentation layer
   HTML templates, CSS, vanilla JavaScript, Mapbox rendering, modals, filters, and page navigation.
2. Application layer
   Flask routes, authentication, session handling, collection management, search endpoints, and image serving.
3. Search layer
   FAISS indexes and keyword indexes built from Redis content at application startup.
4. Storage layer
   Redis databases for objects and user data, plus a static GeoJSON export used by the frontend.
5. Data operations layer
   Harvest, import, enrich, export, and restore scripts under `data_pipeline/`.

### Data Flow

The main runtime data flow is:

1. Redis stores the canonical working dataset for objects and users.
2. On startup, Flask connects to Redis and builds in-memory FAISS and keyword indexes.
3. The frontend requests HTML pages from Flask.
4. The frontend loads [frontend/static/data/redis_export.geojson](/media/data/giacomo.vallasciani/RADICI/frontend/static/data/redis_export.geojson) for map and grid population.
5. Interactive actions such as login, search, collection management, and image operations call Flask endpoints.
6. Images are resolved through Flask image routes rather than by direct filesystem exposure.

### Redis Layout

The application uses two Redis databases by default:

- `DB 10` for cultural objects and related metadata
- `DB 12` for users, authentication-related records, and collections

The canonical dump currently tracked in the repository is:

- [data/redis/radici-redis-dump.rdb](/media/data/giacomo.vallasciani/RADICI/data/redis/radici-redis-dump.rdb)

It is important to understand that this `.rdb` file is not generated by a Python script in the repository. The dump is a Redis snapshot produced by Redis itself after the dataset has been populated and enriched. The Python pipeline scripts prepare and mutate the Redis content; Redis persistence is what ultimately writes the `dump.rdb`-format snapshot.

Typical object fields include:

- `id`
- `title`
- `author`
- `singer`
- `archive`
- `type`
- `date`
- `place`
- `origin`
- `url`
- `image`
- `img_path`

- `model_lora`
- `embeddings`

### Search Architecture

The search stack in [backend/app.py](/media/data/giacomo.vallasciani/RADICI/backend/app.py) has two complementary modes:

- embedding-based similarity search using FAISS
- keyword-oriented search built from tokenized textual fields

At startup:

1. object embeddings are loaded from Redis
2. a FAISS index is created in memory
3. textual fields are tokenized
4. a keyword-to-object mapping is built
5. a second FAISS index for keyword embeddings is created when the sentence-transformer model is available locally

The backend is also tolerant of partial offline conditions:

- if the sentence-transformer model is unavailable locally, keyword embedding search degrades gracefully
- if NLTK resources are unavailable, tokenization falls back to regex-based processing

The frontend is a classic server-rendered application with client-side enhancement.

Key characteristics:

- no Node-based build step is required

### TLS and Local HTTPS Architecture

Local HTTPS support is handled through certificate materials stored under [infrastructure/certs](/media/data/giacomo.vallasciani/RADICI/infrastructure/certs).

There are three categories of TLS-related files:

- CA materials in [infrastructure/certs/ca](/media/data/giacomo.vallasciani/RADICI/infrastructure/certs/ca)
- Flask runtime certificate and key in [infrastructure/certs/flask](/media/data/giacomo.vallasciani/RADICI/infrastructure/certs/flask)
- CSR and generation artifacts in [infrastructure/certs/requests](/media/data/giacomo.vallasciani/RADICI/infrastructure/certs/requests)

The runtime only needs:

- [infrastructure/certs/flask/flask-cert.pem](/media/data/giacomo.vallasciani/RADICI/infrastructure/certs/flask/flask-cert.pem)
- [infrastructure/certs/flask/flask-key.pem](/media/data/giacomo.vallasciani/RADICI/infrastructure/certs/flask/flask-key.pem)

The files in `requests/` are not used during runtime. They are only useful if you want to understand or repeat the certificate-generation workflow.

### Operational Data Architecture

The project distinguishes clearly between:

- runtime data in Redis
- static exported data in `frontend/static/data`
- persistent dump data in `data/redis`
- transformation scripts in `data_pipeline`

This separation is intentional and makes it easier to:

- run the application locally
- restore a clean Redis instance
- update or regenerate static exports
- keep legacy processing scripts away from the application runtime

## Installation and Startup

The repository now provides a full local installer:

```bash
bash install_project.sh
```

This script performs the standard first-time setup for a fresh machine:

- creates or reuses the project virtual environment in `RADICI/`
- upgrades `pip`, `setuptools`, and `wheel`
- installs the active backend runtime dependencies from [backend/requirements.txt](/media/data/giacomo.vallasciani/RADICI/backend/requirements.txt)
- installs the active data-pipeline dependencies from [data_pipeline/requirements.txt](/media/data/giacomo.vallasciani/RADICI/data_pipeline/requirements.txt)
- preloads the NLTK resources used by the application
- creates `.env.local` if it is missing
- ensures the `redis:7.4` Docker image is available
- starts a local Redis container from [data/redis/radici-redis-dump.rdb](/media/data/giacomo.vallasciani/RADICI/data/redis/radici-redis-dump.rdb)

The installer is idempotent and safe to rerun.

Useful installer toggles:

- `INSTALL_PIPELINE_DEPS=false bash install_project.sh`
  Install only the web application runtime.
- `START_REDIS=false bash install_project.sh`
  Prepare the environment without starting the local Redis container.
- `PYTHON_BIN=python3.11 bash install_project.sh`
  Use a specific Python interpreter.

### Installation Prerequisites

Before running the installer, make sure the host machine already has:

- Python 3 with `venv`
- Docker

For the PDF-oriented Benedetti harvesting pipeline, the system also benefits from Poppler tools such as `pdftoppm`, because [harvest_benedetti_media.py](/media/data/giacomo.vallasciani/RADICI/data_pipeline/harvest/harvest_benedetti_media.py) relies on `pdf2image`.

### Dependency Split

Dependencies are now organized more clearly:

- [backend/requirements.txt](/media/data/giacomo.vallasciani/RADICI/backend/requirements.txt)
  Active runtime dependencies for the integrated Flask application.
- [data_pipeline/requirements.txt](/media/data/giacomo.vallasciani/RADICI/data_pipeline/requirements.txt)
  Additional dependencies used by the active data collection, enrichment, and export scripts.

## One-Command Local Startup

If you want the standard local setup without running two commands manually, use:


This script:

- checks whether the local Redis container is already running
- starts Redis from `data/redis/radici-redis-dump.rdb` if needed

- stops the Redis container on exit only if the script started it itself

Use this as the default entry point for local development when you want the full stack.

## Data Pipeline Workflow

Use [data_pipeline](/media/data/giacomo.vallasciani/RADICI/data_pipeline) when you need to:

- collect new source data
- import records into Redis
- enrich records with coordinates or embeddings
- export Redis to CSV or GeoJSON

Do not use `data_pipeline` as the default entry point for normal web application startup. It is an operations and maintenance area, not the runtime application.

### Pipeline Roles

The active pipeline is conceptually split into four phases:

1. Harvest
   Collect raw source material or metadata from external archives.
2. Import
   Create or update object records inside Redis.
3. Enrich
   Fill missing derived fields such as coordinates or embeddings.
4. Export
   Produce frontend-ready or analysis-ready files from Redis.

### Which Script Creates the Redis Dataset

The script that most directly creates the working object dataset in Redis is:

- [data_pipeline/import/import_jsonl_entries_to_redis.py](/media/data/giacomo.vallasciani/RADICI/data_pipeline/import/import_jsonl_entries_to_redis.py)

This importer is the closest thing the current repository has to a dataset seed step for `DB 10`.

Its responsibilities are:

- read a source JSONL export
- keep only records that belong to the RADICI collection scope
- fetch IIIF manifests
- derive one or more image URLs from each manifest
- download image files locally
- normalize selected metadata fields
- create Redis hashes keyed by object or image identifier

The resulting Redis records typically contain fields such as:

- `id`
- `title`
- `author`
- `archive`
- `type`
- `date`
- `place`
- `origin`
- `url`
- `image`
- `img_path`
- `coordinates`
- `embeddings`

At this stage, some fields may still be incomplete or empty, especially derived values such as embeddings and coordinates.

### Import and Enrichment Flow

The current intended flow is:

1. Harvest or obtain source records from archive-specific sources.
2. Import the base records into Redis.
3. Merge external coordinate data when available.
4. Enrich Redis records with derived data.
5. Export Redis to frontend-ready files.
6. Let Redis persist the final state into an `.rdb` snapshot.

### Harvest Scripts

- [data_pipeline/harvest/harvest_classense_items.py](/media/data/giacomo.vallasciani/RADICI/data_pipeline/harvest/harvest_classense_items.py)
  Fetches paginated raw item payloads from the Classense API and writes them to a local JSON file. This is a source acquisition step, not a Redis population step.
- [data_pipeline/harvest/harvest_benedetti_media.py](/media/data/giacomo.vallasciani/RADICI/data_pipeline/harvest/harvest_benedetti_media.py)
  Downloads Benedetti-related PDFs, preview images, and audio files. It prepares local source assets but does not itself create Redis records.

### Import Scripts

- [data_pipeline/import/import_jsonl_entries_to_redis.py](/media/data/giacomo.vallasciani/RADICI/data_pipeline/import/import_jsonl_entries_to_redis.py)
  Creates the initial Redis object hashes in `DB 10` from JSONL source data and IIIF manifests.
- [data_pipeline/import/import_geojson_coordinates_to_redis.py](/media/data/giacomo.vallasciani/RADICI/data_pipeline/import/import_geojson_coordinates_to_redis.py)
  Updates existing Redis records with coordinates from a GeoJSON file. It assumes the records already exist in Redis.

### Enrichment Scripts

- [data_pipeline/enrich/enrich_redis_entries.py](/media/data/giacomo.vallasciani/RADICI/data_pipeline/enrich/enrich_redis_entries.py)
  Iterates through the existing Redis dataset and backfills missing fields:
  - computes image embeddings when an image path exists
  - geocodes places when coordinates are missing
  - stores a `place_coordinates_cache` helper hash in Redis
- [data_pipeline/enrich/generate_image_embeddings.py](/media/data/giacomo.vallasciani/RADICI/data_pipeline/enrich/generate_image_embeddings.py)
  Generates image embeddings from local image files for separate processing workflows.
- [data_pipeline/enrich/enrich_benedetti_coordinates_from_wikidata.py](/media/data/giacomo.vallasciani/RADICI/data_pipeline/enrich/enrich_benedetti_coordinates_from_wikidata.py)
  Looks up coordinates for Benedetti-related records from an external knowledge source.

### Export Scripts

- [data_pipeline/export/export_redis_to_csv_geojson.py](/media/data/giacomo.vallasciani/RADICI/data_pipeline/export/export_redis_to_csv_geojson.py)
  Reads Redis `DB 10` and writes:
  - a CSV export of the record set
  - a GeoJSON export used by the frontend map and grid

This script does not create the Redis dump. It creates file-based exports from Redis.

### How the `.rdb` Snapshot Is Produced

No active Python script in the repository writes `radici-redis-dump.rdb` directly.

The operational model is:

1. Populate Redis with the import scripts.
2. Enrich Redis with the enrichment scripts.
3. Optionally export CSV and GeoJSON.
4. Allow Redis to write a snapshot through its normal persistence mechanism.
5. Preserve or copy that resulting snapshot as the canonical project dump.

In other words:

- Python scripts create and modify the dataset inside Redis.
- Redis creates the `.rdb` snapshot.

### Legacy Dataset Initialization Scripts

Two legacy scripts show older approaches to mass-loading Redis:

- [data_pipeline/legacy/redis_init_legacy.py](/media/data/giacomo.vallasciani/RADICI/data_pipeline/legacy/redis_init_legacy.py)
  Loads a CSV file into Redis and performs a destructive `flushdb()` first.
- [data_pipeline/legacy/add_to_redis.py](/media/data/giacomo.vallasciani/RADICI/data_pipeline/legacy/add_to_redis.py)

These files are useful as historical references but should not be considered the recommended operational path today.

## Important Notes for New Contributors

- the active application entry point is `backend/app.py`
- the frontend is served by Flask and does not require a separate build pipeline
- the Mapbox token is public by design and should be restricted at the Mapbox account level
- the active frontend dataset is the exported GeoJSON in `frontend/static/data`
- many legacy scripts are intentionally preserved but are not production-ready
- some historical scripts contain hardcoded hosts, API keys, or environment assumptions
- the repository contains a very large number of downloaded images under `backend/downloaded_images`

## Known Caveats

- there is still a leftover runtime directory named `.redis-runtime` in the repository root that could not be removed because of external filesystem ownership constraints
- older scripts in `data_pipeline/legacy` should be treated as historical references, not as canonical operational tools
